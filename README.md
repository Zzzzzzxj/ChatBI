# ChatBI

ChatBI 是一个面向企业经营分析场景的自然语言 BI 系统。项目目标是让业务人员可以用接近日常表达的方式查询企业数据，并获得可验证、可解释、可继续分析的结果。

## 项目背景

企业经营分析中，业务人员经常需要围绕收入、利润、费用、客户、产品、区域、时间等维度提出问题。例如：

- 最近一个季度收入表现如何？
- 哪些产品线贡献了主要利润？
- 某个区域的费用是否异常增长？
- 利润下降是收入减少、成本上升，还是费用结构变化导致的？

传统 BI 工具通常依赖固定报表和手工拖拽分析。业务人员一旦提出新的临时问题，往往需要数据分析师理解需求、查询数据库、编写 SQL、整理结果。ChatBI 希望缩短这个链路，让系统承担一部分“理解问题、生成查询、返回结果”的工作。

## 核心问题

ChatBI 不是简单的 Text2SQL。企业数据分析中存在很多直接生成 SQL 难以处理的问题：

- 业务口径：例如“收入”通常有含税收入、不含税收入、人民币折算收入等不同口径。
- Schema 理解：用户说的是业务语言，数据库里是表名、字段名和外键关系。
- 多表关联：收入、成本、客户、产品、汇率、费用等信息往往分散在多张表中。
- 分析拆解：诸如“利润为什么下降”并不是单条 SQL 就能完整回答的问题。
- 结果可信：系统需要让 SQL、查询结果和分析结论可以被检查和追踪。

因此，本项目会从一个最小可运行闭环开始，逐步扩展到更完整的企业级 ChatBI 系统。

## MVP 范围

第一阶段的 MVP 聚焦在自然语言查询到 SQL 执行的最短闭环：

```text
用户提问
  -> 读取数据库 Schema
  -> 构造 Prompt
  -> LLM 生成 SQL
  -> 执行 SQL
  -> 返回 SQL 与查询结果
```

MVP 的目标不是一次性解决所有复杂分析问题，而是先验证以下能力：

- 系统能够接收一个业务问题。
- 系统能够基于数据库结构生成 SQL。
- 系统能够执行 SQL 并返回结构化结果。
- 系统能够暴露生成的 SQL，方便调试和校验。

## 演进路线

项目后续会按企业级 ChatBI 的能力逐步推进：

1. 数据建模：建立制造业经营分析所需的核心业务数据表。
2. Text2SQL：实现从自然语言问题到 SQL 的基础生成能力。
3. 服务化：将命令行流程封装为 API 服务。
4. 前端交互：提供可用的 ChatBI 查询界面。
5. Schema Linking：让系统根据问题选择相关表和字段，降低无关 Schema 干扰。
6. 指标知识库与 RAG：将收入、利润、费用等业务指标口径注入生成过程。
7. 复杂分析 Agent：支持多步骤 SQL 执行、归因分析和结构化报告生成。

## 当前状态

当前仓库处于项目立项和 MVP 蓝图阶段，暂不包含业务代码。后续实现会围绕上述路线逐步提交，每个阶段保持可运行、可验证、可回溯。
