# ChatBI 项目蓝图

## 1. 项目定位

ChatBI 面向企业内部经营分析场景，目标用户是业务人员、经营分析人员和管理者。系统希望降低业务人员使用数据的门槛，让他们可以通过自然语言提出问题，并获得 SQL、查询结果和后续分析依据。

项目初期聚焦制造业和新能源业务域，围绕客户、产品、订单、收入、成本、汇率、费用等数据展开。这个业务域足够贴近真实企业场景，也能覆盖 ChatBI 系统中最典型的难点：业务口径、复杂 Schema、多表 Join、指标解释和分析拆解。

## 2. 企业落地痛点

企业数据分析并不只是“把一句话翻译成 SQL”。真实落地中通常会遇到以下问题：

- 业务语言与数据字段不一致：用户说“营收”“销售额”“利润”，数据库中可能是 `net_amount`、`gross_amount`、`material_cost`、`labor_cost` 等字段。
- 指标口径需要统一：同一个指标在财务、销售、管理报表中可能有不同计算方式，系统必须明确默认口径。
- 数据分散在多张表：一个业务问题可能同时涉及订单、客户、产品、费用、汇率等多类数据。
- 查询结果需要可信：业务人员不能只看到一句回答，还需要看到 SQL 和原始结果，以便判断结果是否可靠。
- 复杂问题需要拆解：例如“利润下降原因”需要先拆指标，再对收入、成本、费用、区域、产品等因素进行对比分析。

这些痛点决定了 ChatBI 需要具备 Query 理解、数据理解和分析能力三层能力。

## 3. 能力拆解

### Query 理解

系统需要识别用户问题中的关键意图，包括：

- 要分析的指标，例如收入、利润、成本、费用、订单量。
- 分析维度，例如时间、区域、客户类型、产品线。
- 过滤条件，例如已完成订单、某个国家、某个产品类别。
- 分析动作，例如查询、排序、对比、归因、趋势分析。

### 数据理解

系统需要理解数据库结构和业务语义，包括：

- 有哪些表和字段。
- 字段代表什么业务含义。
- 表之间如何关联。
- 哪些字段适合做指标，哪些字段适合做维度。
- 哪些业务规则必须在 SQL 生成时被遵守。

### 分析能力

系统最终不应只停留在 SQL 生成，而要支持更复杂的分析任务：

- 将复杂问题拆解成多个可执行子问题。
- 执行多条 SQL 并管理中间结果。
- 对结果做归因、对比和总结。
- 生成结构化分析报告。

## 4. MVP 定义

MVP 的目标是先完成一个最小可运行闭环，而不是一次性构建完整智能分析系统。

### 输入

- 用户自然语言问题。
- 数据库 Schema 信息。
- 必要的生成规则和业务提示。

### 输出

- LLM 生成的 SQL。
- SQL 执行结果。
- 查询是否成功、失败原因和基础调试信息。

### 核心链路

```text
用户提问
  -> Query 预处理
  -> Schema 注入
  -> Prompt 构造
  -> LLM 生成 SQL
  -> SQL 执行
  -> 结果格式化
```

### MVP 模块边界

- Query Parser：负责接收和校验用户问题。
- Prompt Builder：负责组合 Schema、示例和规则。
- LLM Client：负责调用模型生成 SQL。
- Database Client：负责连接数据库并执行 SQL。
- Result Formatter：负责将查询结果整理为可读格式。
- Application Entry：负责串联完整流程。

## 5. 当前阶段暂不实现

为保证第一阶段足够聚焦，以下能力暂不进入 MVP：

- 复杂多轮对话记忆。
- 自动归因报告。
- 多步骤 Agent 执行。
- 权限控制和审计日志。
- 前端页面。
- Schema Linking 和向量检索。
- 指标 RAG 和知识库检索。

这些能力会在基础闭环稳定后逐步加入。

## 6. 后续迭代路线

第一阶段先建立数据环境和 Text2SQL 基础闭环。随后逐步完成：

1. 构建核心业务数据库和模拟数据。
2. 实现基础 Text2SQL 脚本。
3. 引入 Few-shot、业务规则和错误防护。
4. 建立 SQL 执行准确率评估体系。
5. 将系统封装为 FastAPI 服务。
6. 构建可交互前端。
7. 引入 Schema Linking，提升复杂 Schema 下的生成准确率。
8. 引入指标知识库和 RAG，提升业务口径一致性。
9. 引入复杂任务拆解和 Agent 流程，支持更完整的经营分析。

## 7. 项目原则

- 先闭环，再增强：优先完成可运行链路，再逐步优化效果。
- SQL 可见：系统生成的 SQL 必须暴露出来，方便验证。
- 业务口径优先：指标定义和业务规则要显式管理。
- 每次迭代可验证：每个阶段都应有可运行示例或测试结果。
- 项目化演进：仓库历史体现企业级 ChatBI 系统逐步成型的过程。
